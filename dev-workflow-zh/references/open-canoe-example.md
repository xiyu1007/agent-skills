# Open-Canoe：参考示例

本项目完整演示了 `dev-workflow` 工作流的实际应用。它包含两层（Python 上位机 + C 固件），二进制通信协议，真实硬件测试，GUI 模拟测试，以及 30+ 个已追踪 Bug。

## 工作流如何应用

### 1. README.md
最先创建，随功能稳定重写了 3 次。当前简洁明了：功能特性 → 引脚连接 → 前置条件 → 快速开始 → 项目结构。

### 2. doc/SPECIFICATION.md
1300+ 行。系统架构含 App↔Firmware 层的 ASCII 图表。完整的二进制协议定义（帧格式、CRC、命令码）。数据流状态机。实现计划。**每个模块的超时估算**（串口测试：30s，固件烧录：120s，GUI 交互：30s）。

### 3. doc/REQUIREMENTS.md
300+ 行。追踪了 26 个固件 Bug + 10 个应用 Bug。每个 Bug：症状 → 根因 → 修复 → 状态（✅/❌/⚠️）。核心行为规则（正常模式=无 RX，回环模式=RX 匹配 TX）。完整的 UI 控件清单（每个按钮/下拉框/复选框）。跨模块测试规则。文档同步规则。**带明确超时的测试流程**——每个模块的测试步骤在实现前写好，将"完成"定义为契约。

### 4. CLAUDE.md
固件层规则（SPL 风格 CAN、轮询驱动、绝不使用 HAL_CAN_Init）。路径规则（内部用相对路径，外部用环境变量）。活文档检查清单（每次对话后审查三份文档）。**测试超时规则**（每个 subprocess.run 必须带 timeout=，衍生进程的测试必须在 tearDown 中清理，超时是 Bug 而非数字调整）。

### 5. 目录结构
`firmware/` + `open-canoe/` 双层。`test/` 含 5 个测试脚本，**全部带与模块类型匹配的超时**。`tools/` 含部署和清理脚本。

### 6. 测试哲学
`test_gui_full.py` 执行 27 步 CAN 流程测试，使用 `MainWindow()` + `btn.invoke()` + `app.root.update()` 驱动事件循环。通过读取 `_tbl._tree.get_children()` 和 `_log._text.get()` 验证。`test_ui_controls.py` 测试 40 个 UI 控件。**所有测试都有超时和 tearDown 清理**——衍生的应用进程在 tearDown 中杀死，串口在 finally 中关闭，防止资源泄露。

### 7. 配置
单一 `config/defaults.yaml` 存放所有用户设置。`gui/config.py` 存放不可配置的常量（颜色、字体）——它们不是用户设置，不应放在配置文件中。

## 关键教训

- **Bug 追踪表避免了重复调试**：相同的问题（ISR 风暴、HAL_CAN_Init 不可靠、波特率时序错误）被记录了根因，后续调试从已知修复出发，无需重新发现。
- **活文档机制捕获了过时文档**：每次会话后，文档同步检查清单强制审查三份文档。这捕获了过时的架构描述、遗漏的测试标准和失效的 README 功能描述。
- **真实 GUI 点击捕获了仅回调测试遗漏的 Bug**：回环模式下的 ISR 风暴只通过真实事件路径触发（`btn.invoke()` → `app.root.update()`），直接回调调用无法复现。断开/重连的状态损坏也只能通过真实点击序列暴露。
- **测试超时阻止了测试套件挂死**：当串口测试卡住等待永不抵达的字节时，30s 超时将其杀死，tearDown 关闭了串口。没有超时的话，这会阻塞所有后续测试并泄露串口句柄。该卡死被追踪为 REQUIREMENTS.md 中的 Bug #27。
