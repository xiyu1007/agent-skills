# 开发工作流

## ⚠️ 第一步 —— 判断项目阶段（每次对话）

在**每次**对话开始时，检查当前目录：

```
doc/SPECIFICATION.md + doc/REQUIREMENTS.md + CLAUDE.md 是否存在？
  ├─ 否  → 阶段 A：引导新建项目
  └─ 是  → 当前是否属于开发对话（添加功能、修复 Bug、修改代码）？
            ├─ 否  → 无需文档变更，直接处理用户请求
            └─ 是  → 阶段 B：评估当前输入 → 更新文档 → 实现
```

不要跳过此检查。阶段 B 仅在三个条件同时满足时才执行：文档存在、项目处于开发中、且对话涉及代码/功能/Bug 变更。

---

## 阶段 A：引导新建项目

当项目缺少三份核心文档时，按顺序创建它们。

### A.1 语言

在编写任何文档之前，先询问："项目文档使用什么语言？中文 / English / 中英双语？"

双语格式使用锚点导航——每种语言是完整独立的章节，绝不混排：

```markdown
# 项目名称
**EN** · Brief description.  **CN** · 简短描述。
**[English](#english) | [中文](#中文)**
<a name="english"></a>## English
<a name="中文"></a>## 中文
```

### A.2 创建 doc/SPECIFICATION.md

权威开发规格说明书。必须包含以下章节：

- **项目概览**：是什么、为什么
- **系统架构**：ASCII 图表，层次结构，通信路径
- **通信/接口协议**：线格式、API 签名、协议定义
- **组件拆解**：每个模块的职责和接口
- **数据流**：状态机，数据如何流转
- **错误处理策略**：错误的传播和报告方式
- **实现计划**：工作顺序，模块间依赖
- **测试策略**：测试方法论。必须包含每个模块的超时估算。

### A.3 创建 doc/REQUIREMENTS.md

面向测试的文档。必须包含以下章节：

- **项目目标**：1-2 句话
- **功能模块**：表格——模块、功能、测试标准、预期最大测试运行时间
- **测试流程**：带明确超时值的分步测试步骤。这是契约——代码在这些流程通过之前不算"完成"。
- **核心行为规则**：什么必须发生 / 什么绝对不能发生
- **测试方法论**：真实 I/O 模式（见下方通用规则）
- **历史 Bug 追踪表**：编号、症状、根因、修复、状态（✅/❌/⚠️）
- **UI 控件清单**（如有 GUI）：每个控件的预期行为和验证方式
- **测试通过标准**：必须全部通过的通过/失败步骤

### A.4 创建 README.md

最简初版——随功能稳定逐步扩展：

```markdown
# 项目名称
简短一句话描述。
## 快速开始
## 功能特性
```

### A.5 创建目录结构

直接在当前目录中工作——当前目录就是项目根目录。绝不为项目名创建子目录。

**单层项目：**

```
├── README.md
├── CLAUDE.md
├── doc/SPECIFICATION.md, REQUIREMENTS.md
├── config/defaults.yaml
├── data/          # 运行时生成，不提交
├── src/
├── test/          # 所有测试，每个测试必须带超时
└── tools/         # 部署/清理脚本
```

**多层项目**（如 固件 + 桌面应用）：

```
├── doc/           # 3 份文档覆盖所有层 —— 绝不分层
├── firmware/      # 层，可选 config/、data/
├── app/           # 层，可选 config/、data/
├── test/          # 共享，所有测试带超时
└── tools/         # 共享
```

**规则**：恰好 3 份文档（README + SPECIFICATION + REQUIREMENTS），绝不分层。`config/` = 需要提交的配置文件。`data/` = 不提交的运行时文件。

### A.6 创建 CLAUDE.md

使用本技能末尾的模板。文档同步块和超时规则放在文件最顶部。

## 阶段 B：持续开发 —— 评估并更新文档

仅当**三个条件同时满足**时才进入：(1) 三份核心文档已存在，(2) 项目处于活跃开发中，(3) 对话涉及代码/功能/Bug 变更。满足时，**在编写任何代码之前**，根据用户输入评估每份文档：

| 文档                       | 自问                                                           |
| -------------------------- | -------------------------------------------------------------- |
| **SPECIFICATION.md** | 架构是否变化？模块增减？接口变更？超时估算更新？               |
| **REQUIREMENTS.md**  | 新的行为规则？需要新的测试流程？Bug 需追踪？已有测试缺少超时？ |
| **README.md**        | 新功能是否稳定到可以写入文档？快速开始有变化？                 |

提取隐式需求：如果用户说"我希望 X 能做 Y"或"X 应该像 Y 那样工作"，立即将其写入 REQUIREMENTS.md。

**只要修改了代码，文档审查就是强制性的。不可跳过。**

### B.1 添加功能/模块 —— 测试流程优先

**先写测试流程，再写代码。** 顺序：

1. 更新 SPECIFICATION.md —— 模块描述、架构、接口、超时估算
2. 更新 REQUIREMENTS.md —— 添加带分步步骤和超时值的测试流程。添加行为规则。
3. 与用户确认测试流程是否正确
4. 然后才实现功能
5. 运行已有测试（回归检查，全部带超时）
6. 执行 REQUIREMENTS.md 中编写的测试流程 —— 验证真实输出
7. 修复 Bug，更新 Bug 追踪表

测试流程就是契约——它在实现开始之前就定义了"完成"的含义。

### B.2 Bug 修复工作流

1. 用测试复现（带超时）
2. 修复代码
3. 用测试验证
4. 记录到 REQUIREMENTS.md Bug 追踪表：症状 → 根因 → 修复 → 状态 ✅

## 通用规则（两个阶段均适用）

### ⚠️ 测试超时 —— 严重

**每个测试必须有超时。没有例外。** 挂起的测试是严重 Bug —— 它会阻塞开发流程并泄露资源。

**按模块类型的超时估算：**

| 模块类型        | 最大超时 | 原因            |
| --------------- | -------- | --------------- |
| 纯函数/单元测试 | 5s       | 近乎瞬时完成    |
| 文件 I/O        | 10s      | 磁盘操作很快    |
| CLI 子进程      | 15s      | 进程创建 + 执行 |
| GUI 交互测试    | 30s      | 事件循环 + 渲染 |
| 串口/硬件测试   | 30s      | 物理设备响应    |
| 网络/API 测试   | 20s      | HTTP 往返       |
| 应用完整流程    | 30s      | 完整生命周期    |
| 固件烧录 + 测试 | 30s      | 硬件编程时间    |

**实现方式：**

```python
# pytest
@pytest.mark.timeout(30)
def test_serial(): ...

# subprocess —— 始终设置 timeout
subprocess.run(['python', 'app.py'], capture_output=True, text=True, timeout=30)

# serial —— 单次读取超时
ser = serial.Serial('COM7', 115200, timeout=2)
```

**清理 —— 超时时杀死孤儿进程：**

```python
class AppTester:
    def setUp(self):
        self.proc = subprocess.Popen(['python', 'app.py'], stdout=subprocess.PIPE, stderr=subprocess.PIPE)
        time.sleep(2)

    def tearDown(self):
        """始终杀死衍生的进程，即使超时或出错。"""
        if self.proc and self.proc.poll() is None:
            self.proc.terminate()
            try:
                self.proc.wait(timeout=5)
            except subprocess.TimeoutExpired:
                self.proc.kill()
                self.proc.wait()
```

串口：始终在 `finally` 中关闭 —— `if ser and ser.is_open: ser.close()`

**如果测试超时，首先当作 Bug 处理。** 调查死锁、未释放资源、无响应依赖。只有确实需要更长时间时才增加超时——并记录原因。

**提交前超时检查清单：**

- [ ] 每个 `subprocess.run()` 都有 `timeout=`
- [ ] 每个 `requests.get/post()` 都有 `timeout=`
- [ ] 每个 `serial.Serial()` 都有 `timeout=`
- [ ] 衍生进程的测试在 `tearDown` / `finally` 中有清理
- [ ] 超时值与模块类型匹配

### ❌ 禁止：虚假测试

| 虚假模式                              | 为什么失败                             |
| ------------------------------------- | -------------------------------------- |
| "我审查了代码，逻辑看起来正确"        | 代码可以看起来完美但在运行时失败       |
| "直接调用函数返回了正确的值"          | 直接测试回调跳过了真实的触发路径       |
| "响应应该包含……"（猜测）            | 假设输出而不是读取实际输出             |
| "如果设置 X，那么 Y 应该发生"（假设） | 理论与实践——硬件、时序、竞态条件不同 |

### ✅ 必须：真实 I/O 验证

每个测试必须与真实系统边界交互：

- **GUI**：`btn.invoke()` → `app.root.update()` → 读取控件状态（`tree.get_children()`、`text.get()`）
- **CLI**：`subprocess.run()` 带 `timeout=`，检查 `returncode` + `stdout` + `stderr`
- **API**：真实 HTTP 调用用 `requests`，带 `timeout=`，检查 `status_code` + 响应体
- **串口/硬件**：`serial.Serial()` 带 `timeout=`，读取实际字节，验证 hex 内容
- **断言带证据**：每次失败都打印预期值 vs 实际值

## 示例：阶段 B 实践

> 用户："应用应该在回环模式下工作，发送的消息作为 RX 返回"

→ SPECIFICATION.md：在架构中添加回环模式
→ REQUIREMENTS.md：添加行为规则（回环模式：TX+RX 行出现，RX 的 ID/数据与 TX 匹配；正常模式：只有 TX 行，绝对没有 RX 行；模式切换不得产生残留帧）

> 用户："添加一个串口监控模块来记录收到的字节"

→ SPECIFICATION.md：添加模块描述、接口协议
→ REQUIREMENTS.md：添加带 30s 超时的测试流程 —— 打开串口、发送已知字节、验证监控模块捕获到它们、在 finally 中关闭串口
→ 然后才实现

> 用户："删除按钮在移除行后应该清除选中状态"

→ REQUIREMENTS.md：添加带超时的测试流程，添加到 UI 控件清单
→ SPECIFICATION.md：仅当涉及架构变更时（UI 行为变更通常不涉及）

## CLAUDE.md 模板

```markdown
# CLAUDE.md — 项目名称

## ⚠️ 请先阅读 —— 文档同步（不可跳过）

每次对话后，回复之前：

1. 审查 `doc/SPECIFICATION.md` —— 架构变化？新增模块？接口变更？
2. 审查 `doc/REQUIREMENTS.md` —— 新 Bug？行为变更？新测试流程？缺少超时？
3. 审查 `README.md` —— 新功能是否稳定到可以写入文档？
4. 如果用户说了"我希望 X 能做 Y"或"添加模块 Z"，首先将其提取为测试流程写入 REQUIREMENTS.md

**只要修改了代码，就必须检查文档。不可跳过。**

## ⚠️ 测试超时规则（不可跳过）

- 每个测试必须有与其模块类型匹配的超时
- 衍生进程的测试必须在 tearDown/finally 中清理
- 如果测试超时：先作为 Bug 调查，再考虑增加超时
- 超时无清理 = 孤儿进程 = 资源泄露 = 严重 Bug
- 提交前：验证每个 subprocess.run/requests/serial.Serial 调用都有 timeout=
```
